# TOOLS.md — Tool Index

> **⚠️ This is a STRUCTURAL REFERENCE, not the live TOOLS.md.**
>
> `/bootstrap` Step 6c **regenerates the root `TOOLS.md` from live probe results** (CLI `--version` / `auth status` calls, deferred-tools list for HTTP MCPs, env-var checks for stdio MCPs). The status flags in the regenerated file (✅ / ⏳ / ⚠️ / ❌ / ℹ️) trace back to real probe output — never to what's typed below.
>
> This template exists so a fork user can see **what shape** the root TOOLS.md should have. Do NOT edit this file to claim a tool is ready — edit the ROOT `TOOLS.md` after `/bootstrap` writes it, or re-run `/bootstrap` (delete the `setup_completed:` line in `CLAUDE.md` first) to regenerate from fresh probes.

**Status legend (used by the live TOOLS.md, NOT below):**
- ✅ connected / authed / on PATH
- ⏳ configured but not authorized for this workspace (HTTP MCPs typically — run `/mcp`)
- ⚠️ installed but auth broken / partial state — fix command surfaced
- ❌ not installed — install hint surfaced
- ℹ️ unverified — probe couldn't determine state, manual check required

## Platform formatting

- **WhatsApp:** No markdown tables, no headers. Use `*bold*` and bullet lists.
- **Discord:** Wrap multiple links in `<>` to suppress embeds.
- **Slack:** plain markdown in messages; canvases support richer formatting. Avoid `#` headers in regular messages — use `*bold*`.

## Discipline: every entry needs a probe

Adding an entry to TOOLS.md? Back it with a **verification probe** — a command output, a test invocation, a Bash check. Vibes-based entries drift, and the next person reading this file (including future-you) will trust them.
